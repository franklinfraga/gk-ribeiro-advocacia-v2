§ 1. The two cultures of concurrency

There are two ways to write concurrent programs in Rust, and they have almost nothing in common. The first inherits from C: you spawn operating-system threads, share memory through Mutex<T> and Arc<T>, and reason about correctness the way you reason about a database transaction — by holding locks in a fixed order and hoping nobody reorders them six months from now.

The second inherits from Erlang: you create lightweight tasks, you send messages between them, and you never share mutable state at all. The compiler can prove this; the runtime can guarantee it.

Most teams pick one and stick with it. A few try to use both in the same codebase, and those teams usually end up writing two programs that happen to share a Cargo.toml. This chapter is about the first culture, because it is the harder one to learn, and the one the borrow checker was specifically designed to make safe.

§ 2. Threads, and what they cost

A thread in Rust costs about eight kilobytes of stack on Linux, plus whatever the kernel allocates for its bookkeeping. That number is small enough that you can spawn thousands of them, and large enough that you should not spawn millions.

The cost you actually feel, though, is not the memory — it is the coordination. Every shared piece of state needs a lock, every lock needs a discipline, and every discipline is one rename away from a deadlock.

Every lock needs a discipline. Every discipline is one rename away from a deadlock. — the first law of concurrent Rust

Here is the smallest interesting example: ten threads incrementing a shared counter.

use std::thread; use std::sync::{Arc, Mutex}; let counter = Arc::new(Mutex::new(0)); let mut handles = vec![]; for _ in 0..10 { let counter = Arc::clone(&counter); handles.push(thread::spawn(move || { let mut num = counter.lock().unwrap(); *num += 1; })); } for handle in handles { handle.join().unwrap(); } println!("Result: {}", *counter.lock().unwrap());

The output is 10, every time. That is the whole point. Swap Mutex for RefCell and the program will not compile; the borrow checker has your back. Drop the Arc and the program will not compile either; lifetime analysis has your back. The only thing the compiler cannot do is stop you from writing a deadlock — that one is on you.

If the playground feels pedestrian — three jobs, three results — that is the point. The interesting line is the one you do not see: no thread ever read another thread's half-written data. In Rust that is a compile-time guarantee. In JavaScript it is a single-threaded runtime. Both work; only one scales to eight cores without you noticing.


        
§ 4. From channels to scoped threads

        
Channels work, but for short-lived threads they are more machinery than the job needs. Rust 1.63 introduced scoped threads, which let you borrow from the parent scope without channels or Arc. Click apply below to watch the refactor happen.


        

          

            

              Refactor · Rust 1.63+
              From channels to scoped threads — same semantics, half the code.
            

            
          

          

            
let (tx, rx) = mpsc::channel();

            
let h = thread::spawn(move || {

            
    tx.send("hello").unwrap();

            
});

            
println!("{}", rx.recv().unwrap());

            
h.join().unwrap();

            
thread::scope(|s| {

            
    let h = s.spawn(|| "hello");

            
    println!("{}", h.join().unwrap());

            
});

            
// both versions print: hello

          

        


        
The right side borrows "hello" directly from the parent scope. The compiler can prove the threads will not outlive the scope, so no channel, no Arc, no clone. Six lines become four, and the type system did the audit for you.


        
§ 5. When to use what

        
After six years of writing concurrent Rust in production, my rule of thumb is small enough to fit on a sticky note:

        
If the work is independent and CPU-bound, use rayon. If the work is I/O-bound, use async. If you are sharing a small piece of state across threads, use Mutex. If you are sharing a lot of state, redesign until you aren't.

        
That last rule is the most important one in this chapter. The borrow checker will not let you write a data race, but it will happily let you write a program that is correct and slow — a program that spends half its runtime acquiring locks it didn't need, and the other half waiting on channels it didn't need either. Concurrency is not a feature you add. It is a structure you choose, and the best choice is almost always the smallest one.

        
Chapter 6 takes the other culture — async — and shows why it is not, as the docs sometimes suggest, just "threads but cheaper."
